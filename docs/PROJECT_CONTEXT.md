# Project Context

## What This Project Is
This repository contains a small two-person hydration tracking application built with Next.js App Router, React, TypeScript, Tailwind CSS v4, and Supabase.

The product concept is a shared water-tracking experience for two named participants. The implemented UX frames hydration as a couple activity with daily logging, weekly comparison, monthly comparison, and a reward mechanic for the monthly winner.

## What Was Verified
- The application code lives in `water-app/`.
- The workspace root also contains a separate `package.json`, `package-lock.json`, and `node_modules/`, but those do not define the runnable Next.js app.
- `water-app/` is its own project root with its own `.git`, `.env.local`, `package.json`, `node_modules/`, `.next`, and source tree.
- Supabase is used directly from the browser via the public URL and public anon key.
- No server routes, server actions, API route handlers, middleware, tests, migrations, schema files, or seed files are present.

## Business Purpose
Confirmed from the UI copy and data model usage:
- Track daily water intake for one logged-in user.
- Compare two paired users over a week.
- Roll weekly winners into a monthly result.
- Present hydration as a gentle, emotionally warm, shared ritual.

Inferred from code usage rather than explicit product docs:
- The target audience appears to be exactly two named users rather than an open multi-user consumer app.
- The app is designed more like a private shared ritual than a generic productivity tracker.

## Folder Structure

### App Root
- `.env.local`: committed local environment variables for Supabase.
- `.gitignore`: standard Next.js ignore rules.
- `README.md`: default `create-next-app` README, not project-specific.
- `water-app-source.zip`: archive copy of the same project source.
- `googlestitchfiles/stitch-login.html`: separate static HTML prototype for the login screen.

### Source
- `src/app/`: App Router pages, root layout, global CSS, favicon.
- `src/components/`: reusable UI wrappers used across pages.
- `src/lib/`: Supabase client, auth helper, user hook, and one participant constant.

### Static Assets
- `public/*.svg`: default Next.js example assets. No application code imports them.
- `src/app/favicon.ico`: binary favicon asset.

## Completed Features
- Login form that converts `name + 4-digit pin` into Supabase email/password credentials.
- Session-aware user lookup via `useUser()`.
- Daily water total load and save to Supabase `daily_water`.
- Weekly per-day load from Supabase `daily_water`.
- Weekly partner lookup from Supabase `user_pairs`.
- Weekly result persistence to Supabase `weekly_results` after the week ends.
- Monthly weekly-result load from Supabase `weekly_results`.
- Monthly result persistence to Supabase `monthly_results` after the month ends.
- Bottom navigation across `today`, `week`, `month`, and `settings`.
- Logout via Supabase auth sign-out.

## Partially Implemented Features
- Weekly lock/reveal concept exists in the UI, but much of the detailed weekly card data is still hard-coded.
- Monthly winner reward UI exists, but the wish text is not persisted or sent anywhere.
- Settings screen exists, but nearly all controls are presentational only.
- Setup passcode screen exists, but no logic is attached.

## Missing Features
- User registration or passcode setup flow backed by Supabase auth.
- Route protection or auth redirects.
- Real profile/avatar management.
- Reminder scheduling and delivery.
- Passcode update implementation.
- Weekly navigation to previous/next weeks.
- Monthly history browsing.
- Wish submission persistence or notification delivery.
- Validation against negative water totals or unrealistic input.
- Loading states, optimistic updates, and robust error UX.
- Tests, monitoring, analytics, and deployment docs.

## Important Constraints From The Current Code
- User names are effectively hard-coded to `Abdul` and `Shaima` for friendly labeling.
- Identity pairing depends on the `user_pairs` table and current authenticated user ID.
- The app assumes client-side access to Supabase tables through public credentials and likely RLS.
- There is no typed database client or schema-generated TypeScript.
- There is no central domain layer; page components embed data access directly.

## Artifacts That Are Not Product Architecture
- `water-app/.next/`: generated build/dev output.
- `water-app/node_modules/`: dependency install.
- workspace-root `node_modules/`: separate dependency install unrelated to the app runtime.
- `public/*.svg`: default starter assets not wired into the app.

## Explicit Gaps
- No written product requirements exist in the repository.
- No schema SQL, migration history, or RLS policies are present.
- No API contracts are documented in code comments or external docs.
- No explanation exists for why the workspace root has a second package manifest.
